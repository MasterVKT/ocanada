Analyse stratégique

Analyse SWOT de l'entreprise

Cette analyse révèle que malgré un marché à fort potentiel, Ô Canada reste limitée par une gestion administrative entièrement manuelle — ce qui justifie pleinement notre projet.

💪 Forces

  • Forte demande d'immigration vers le Canada au Cameroun
  • Expertise légale interne pour les dossiers complexes
  • Structure organisationnelle claire et fonctionnelle
  • Volonté de professionnalisation des processus internes

⚠️ Faiblesses

  • Seulement 13 collaborateurs — capacité limitée
  • Absence de système formalisé de gestion RH
  • Dépendance aux pratiques manuelles (WhatsApp, oral)
  • Marque encore peu connue face à la concurrence

🚀 Opportunités

  • Augmentation constante des demandes d'immigration Canada
  • Différenciation par l'innovation technologique
  • Élargissement vers d'autres destinations
  • Partenariats potentiels avec universités canadiennes

⚡ Menaces

  • Nombreuses agences concurrentes à Douala
  • Modifications fréquentes des lois d'immigration
  • Risque de non-conformité au Code du Travail camerounais
  • Risque de départ de collaborateurs clés

L'analyse révèle un risque croissant de non-conformité juridique (Code du Travail, cadre OHADA) lié à l'absence de données RH structurées — un risque que notre système vient directement neutraliser.

Chapitre II — Diagnostic

Le système existant : quatre dysfonctionnements majeurs

🔴 Dysfonctionnement 1

Absence de traçabilité des présences
Aucun registre ni outil numérique. Les heures d'arrivée/départ sont communiquées oralement ou par WhatsApp, sans archivage ni consolidation. Calcul des heures travaillées impossible.

🔴 Dysfonctionnement 2

Gestion désorganisée des congés
Demandes transmises verbalement, sans formulaire ni vérification de solde. Risque réel de violation du Code du Travail camerounais et du cadre OHADA.

🔴 Dysfonctionnement 3

Absence de gestion des visiteurs
Aucune consignation des entrées/sorties, aucun badge, aucun historique. Exposition à des risques de sécurité et de confidentialité des données clients.

🔴 Dysfonctionnement 4

Absence d'indicateurs RH
Sans données centralisées, la direction ne dispose d'aucune base fiable pour piloter la présence, anticiper les besoins ou estimer le coût de l'absentéisme.

Conception — Modèle Conceptuel des Traitements

MCT — Pointage Kiosque & Gestion des Congés

PROCESSUS 1 — POINTAGE KIOSQUE
⚡ Employé saisit matricule + PIN → clique "Pointer"
↓ Synchronisation ET (identifiant ET PIN présents)
1
Vérifier adresse IP dans config_systeme (ip_kiosque_autorisees)
2
Rechercher employé par matricule — vérifier statut = actif
3
Comparer PIN (bcrypt) — 3 erreurs = blocage 10 min sur ce terminal
4
Vérifier fenêtre horaire (arrivée 06h00-10h30 / départ 15h00-21h00)
5
Vérifier cohérence (doublon arrivée / départ sans arrivée)
6
INSERT presences + calcul statut_arrivee + INSERT audit_log
✅ Succès
Confirmation 4s : photo, nom, heure, statut
❌ Échec
Message erreur spécifique — aucune insertion en base
PROCESSUS 2 — GESTION DES CONGÉS
⚡ Employé remplit formulaire de demande (type, dates, motif)
↓ Synchronisation ET (tous champs obligatoires présents)
1
Calculer jours ouvrables (excl. week-ends + jours fériés camerounais)
2
Vérifier solde disponible ≥ jours demandés (si congé annuel)
3
Détecter chevauchements avec demandes existantes approuvées
4A
ADMIN Approuve → UPDATE demande + UPDATE soldes (transaction MySQL atomique)
4B
ADMIN Refuse → UPDATE statut=refusee + commentaire_admin REQUIS
5
INSERT notification → Employé + INSERT audit_log (TRAITEMENT_CONGE)
⏰ Cron 23h59 : NOTIF_CONGE_ATTENTE_48H → ADMIN si demande en attente depuis plus de 48h sans traitement

MCT Visiteurs · Use Case — 4 acteurs

MCT — PROCESSUS 3 — VISITEURS
⚡ Visiteur se présente — Agent ouvre le formulaire
1
Saisir identité visiteur : nom, type pièce, N° pièce, téléphone
2
Motif visite + personne à voir + capture webcam (optionnel)
3

4

5
Durée > seuil → NOTIF_VISITEUR_LONG → ADMIN + AGENT
6
Agent clique sortie → UPDATE heure_sortie + duree_visite_minutes
⏰ Cron 23h59 : UPDATE statut=sorti + note "Sortie non enregistrée" pour visites encore ouvertes

11 mesures de sécurité implémentées

Protection XSShtmlspecialchars() sur toutes les sorties HTML — couche Vue
🎟️
Tokens CSRFFormulaires POST + endpoints AJAX (header X-CSRF-Token)

Hachage bcrypt (coût=12)Mots de passe + PIN kiosque — jamais stockés en clair
🚫
Brute Force LoginBlocage 15 min après 5 tentatives échouées, journalisé
📍
Restriction IP KiosquePointage possible uniquement depuis l'IP du terminal physique (config_systeme)
👮
RBAC complet (AuthFilter + RoleFilter)Appliqués sur tous les groupes de routes — redirection 403 + audit si refus

Fichiers hors racine webDocuments RH stockés avec noms UUID, type MIME vérifié côté serveur
🤖
Clé API IA sécuriséeconfig_systeme BDD — jamais transmise côté client
📜
Journal d'Audit immuableAuditLogModel : INSERT uniquement — aucun UPDATE/DELETE possible — 21 types d'événements
💰
Données confidentiellesSalaires journaliers visibles ADMIN uniquement — filtrage SQL par rôle
